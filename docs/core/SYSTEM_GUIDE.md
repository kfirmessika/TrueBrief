# 🏗️ The TrueBrief Development System
> **Purpose:** A reusable, enterprise-grade workflow for AI-assisted solo development.

## 📐 The Three-Tier Planning Architecture

### 1. The Roadmap (`docs/roadmap.md`)
- **Role:** The High-Level Dashboard.
- **Format:** 1-line-per-step title. Simple `[ ]` / `[x]` status.
- **Why:** Extremely cheap for the AI to read and update. Gives the user an instant view of project progress without token bloat.

### 2. The Execution Blueprint (`docs/core/EXECUTION_PLAN.md`)
- **Role:** The Strategy Map.
- **Format:** Static document.
- **Optimization:** Generated by a high-level model (Opus) using the `.ai/refs/EXECUTION_PLANNER_SPEC.md`.
- **Contains:**
    - **Execution Runs:** Groups of tasks batched by Model and Context (same files).
    - **Context Clusters:** Groups of tasks that touch the same files/ideas.
    - **Dependency Graph:** Logic of what must be built first.
- **Why:** Prevents "context switching." Maximizes API call value by doing multiple related tasks in one session.

### 3. The Blueprints (`docs/blueprints/phase_N.md`)
- **Role:** The Engineering Blueprints.
- **Format:** Detailed technical blocks for every step.
- **Contains:** Design logic, Whitelisted files (Reads/Touches), and Acceptance Criteria.
- **Why:** This is the "Recipe" the AI follows during the Build phase. It is read surgically (one section at a time) to save tokens.

---

## 🛰️ The Execution Loop
1. **SELECT:** Pick a Cluster from the `EXECUTION_PLAN`.
2. **BOOT:** Tell the AI to read `.ai/BOOT.md` and start the steps in that cluster.
3. **PLAN:** Use the assigned **Plan Model** to write a `STEP_SPEC.md`.
4. **BUILD:** Use the **Build Model** to write the code.
5. **TEST:** Use the **Test Models** to verify and check off the `[ ]` boxes in the Phase Plan.
6. **UPDATE:** Mark the step as `[x]` in the Slim Roadmap.
7. **CLEAR:** Run `/clear` and move to the next step.
